Prompts as software artifacts

aikido

AI-native knowledge & decision operations for startup teams. Capture what happens, aggregate it weekly, and transpile reusable AI agents from your own knowledge base — all as flat files you can diff, review, and commit.

Get started → View on GitHub

Install & run in 30 seconds

# install
$ pip install -e .

# scaffold a knowledge base (+ git)
$ aikido init my-company
$ cd my-company

# capture today, aggregate the week, build your agents
$ aikido daily
$ aikido weekly
$ aikido build_agent founder_weekly_review --week 2026-W29

The built artifact lands in 06_dist/founder_weekly_review_2026-W29.md — a flat, self-contained Markdown system prompt with every include resolved and its contract validated.

Why aikido

🧱 Prompts as artifacts

System prompts are assembled from modular Markdown, Jinja2 templates, and JSON contracts — not pasted into a chat box.

⚙️ A real transpiler

Includes, extends, contract outputs, and output constraints are validated at build time across three staged checks.

📓 Daily → weekly cadence

Guided daily capture auto-tags entries and aggregates into decisions, blockers, customer signals, and takeaways.

📄 Everything is a file

No database, no hidden state. Your knowledge base, contracts, config, and artifacts are all plain text on disk.

✅ Contracts & validation

Each agent has a JSON contract with a definition of done and constraints. aikido validate keeps it honest.

🔌 Sync the sources

Pull from Notion, Linear, Slack, and Stripe into the knowledge base, then transpile fresh agents from it.

The build pipeline

Every agent is compiled by AikidoTranspiler through three stages. A failure stops the build and reports exactly where.

1

pre_transpile

Every static {% include %} / {% extends %} target exists, and every contract output id is referenced in the template.

2

transpile

Render the Jinja2 template against the JSON contract plus your variables (week, focus area, …).

3

post_transpile

The rendered output satisfies contract constraints — word counts, no generic filler — before it's written to 06_dist/.

The knowledge tree

aikido init scaffolds this layout with working defaults.

00_control_plane/ # founder identity, tone, safety guardrails 01_weekly_cadence/ # per-week daily logs + aggregated summaries 02_domain_knowledge/ # product + go-to-market knowledge 03_support_tickets/ # support themes 04_contracts/ # JSON contracts (definition of done, constraints) 05_agents/ # Jinja2 prompt templates (*.prompt.md) 06_dist/ # built flat-Markdown artifacts 07_archive/ # compressed old weeks
File-first by design. aikido never hides state in a database or a service. This documentation site is built the same way — hand-authored static HTML5, no build step — so it deploys straight from /docs on GitHub Pages and stays readable forever.
