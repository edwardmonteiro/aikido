Concepts

aikido borrows the discipline of software engineering — modularity, contracts, build-time validation — and applies it to AI prompts and operational knowledge.

Prompts as software artifacts

A good system prompt isn't a paragraph you paste into a chat box. It's assembled from reusable parts: who you are (the control plane), what happened recently (weekly cadence), what you know (domain knowledge), and what "done" means (a contract). aikido keeps each of those as its own file and compiles them together.

The payoff is the same as with source code: you can diff a change, review it, roll it back, and reason about why an agent produced what it did — because its entire input is a flat file on disk.

The transpiler pipeline

AikidoTranspiler.build() runs three stages. Each is a gate: a failure stops the build and reports the stage and the specific errors, so you never ship a half-resolved prompt.

1

pre_transpile — static checks

Parse the template's Jinja2 AST. Confirm every constant {% include %} and {% extends %} target exists on disk, and that every output id declared in the contract's definition of done is actually referenced in the template. Dynamic includes (built from expressions) are skipped here and resolved at render time.

2

transpile — render

Render the template with Jinja2 (trim_blocks + lstrip_blocks) against the loaded contract and your variables — week, focus_area, and anything else you pass. A custom load_json filter/global lets templates read contracts directly.

3

post_transpile — constraint checks

Run the rendered output against the contract's constraints — word_count < N limits and bans on generic filler ("leverage", "synergy", …). On success the artifact is written to 06_dist/{agent}_{week}.md.

Contracts

A contract is a JSON file in 04_contracts/ that pins down what an agent must produce. It has three parts:

  • objective — the job, the user, the context, and the "success feeling."
  • definition_of_done — the required outputs (each with a format and pass/fail criteria) and the constraints the whole output must satisfy.
  • validation — automated auto_checks and open human_review questions.
{
  "job_id": "weekly-founder-review",
  "objective": { "job": "...", "user": "Founder/CEO", ... },
  "definition_of_done": {
    "outputs": [
      { "id": "next_week_priorities", "format": "numbered_list",
        "count": "3", "criteria": "Each must reference a signal or blocker" }
    ],
    "constraints": ["Total output < 500 words", "No generic advice"]
  },
  "validation": {
    "auto_checks": ["word_count < 500", "all_output_ids_present"],
    "human_review": ["Does this feel like me?"]
  }
}

Because the transpiler checks the template against the contract at build time, a contract and its template can't silently drift apart.

The control plane

00_control_plane/ holds the stable facts that every agent should inherit — founder_identity.md, tone_voice.md, and safety_guardrails.md. Templates include them once, so a change to your voice or your guardrails propagates to every agent on the next build.

Templates & inheritance

Agents live in 05_agents/ as *.prompt.md files. A shared _base.prompt.md lays out the skeleton — identity, job, definition of done, constraints, validation — and defines empty context and task blocks. Concrete agents {% extends %} the base and fill those blocks, pulling in the right week's cadence and, optionally, focus-area domain knowledge.

Gotcha worth knowing. In Jinja, output.values resolves to a dict's built-in .values() method rather than a missing key — so aikido's base template uses output.get('values'). It's the kind of edge case build-time validation is designed to catch.

File-first, always

There is no database and no server. Your knowledge base, contracts, config (.aikido/config.yaml), and built artifacts are all plain text. That makes aikido trivially version-controllable, reviewable in a pull request, and portable — and it's why this documentation site is itself just hand-authored static files.
