Knowledge structure

Everything is a file. The knowledge base on the left composes — through the transpiler — into flat Markdown artifacts on the right. This is the whole system on one page.

build / transform composed into the artifact lifecycle (archive)

How to read it

  • Left — the knowledge base. Six kinds of files, each in its own numbered directory. You edit these by hand or via aikido daily, weekly, update, and sync.
  • Daily → rollups. aikido weekly aggregates the week's *_daily.md logs into decisions, blockers, customer_signals, and daily_summary.
  • Templates inherit. Each agent {% extends %} a shared _base.prompt.md, so identity, definition-of-done, and constraints are defined once.
  • Center — the transpiler. It resolves every include, renders the template against the contract, and checks constraints in three gated stages.
  • Right — the artifacts. Successful builds land in 06_dist/ as flat, self-contained Markdown. Old weeks compress into 07_archive/.

This diagram is a hand-authored SVG — no charting library, no build step — so it renders identically everywhere and diffs like any other file, the same way aikido treats prompts.
